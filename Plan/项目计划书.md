# 心理健康监测系统项目计划书

## 一、项目基本信息

### Q1：这个项目要做成什么类型？

本项目定位为一个面向课程设计 / 毕业设计的心理健康监测系统原型。

系统采用前后端分离架构，围绕"学生心理状态辅助监测"展开，核心功能包括用户登录注册、人脸情绪识别（图片+视频）、心理状态记录、风险预警、趋势分析和干预建议。

本系统不作为医学诊断工具，所有情绪识别和风险预警结果仅用于辅助观察和管理参考。

### Q2：系统主要面向哪些用户？

系统暂定包含两类角色：

1. 学生
   - 注册和登录系统
   - 拍照或上传人脸图片进行情绪识别
   - 查看自己的情绪识别记录、心理趋势和系统建议

2. 管理员
   - 登录后台管理端
   - 查看学生列表
   - 查看学生情绪识别记录
   - 查看风险预警列表
   - 使用视频分析功能，逐帧分析学生视频情绪变化
   - 查看基础统计和趋势图

第一版不加入心理咨询师角色，避免权限设计过于复杂。

## 二、项目建设目标

### Q3：为什么要做这个系统？

学生心理健康问题具有一定隐蔽性，传统人工观察方式难以及时发现异常状态。

本项目希望通过人脸情绪识别（图片+视频帧）、长期记录和规则预警，构建一个简单可用的心理健康辅助监测系统，使管理员能够及时关注存在潜在风险的学生。

### Q4：项目最终要达到什么效果？

项目第一版需要形成一个完整可演示的系统闭环：

1. 学生能够注册、登录。
2. 学生能够通过拍照或上传图片完成情绪识别。
3. 系统能够保存情绪识别结果。
4. 系统能够根据情绪记录生成风险等级。
5. 管理员能够通过视频逐帧分析，观察学生情绪变化趋势并获取心理评估。
6. 管理员能够查看学生记录和风险预警。
7. 系统能够展示基础趋势图和统计图。

## 四、技术选型

### Q6：前端使用什么技术？

前端推荐使用：

1. Vue 3
2. Vite
3. Element Plus
4. ECharts
5. Axios
6. Vue Router
7. Pinia

选择原因：

1. Vue 3 上手快，适合课程和毕设项目。
2. Element Plus 适合快速开发登录页、表格、表单和后台管理页面。
3. ECharts 适合实现情绪趋势图、风险分布图和统计大屏。
4. 前端整体复杂度适中，方便后续维护。

### Q7：后端使用什么技术？

后端推荐使用：

1. Python
2. FastAPI
3. SQLAlchemy
4. Pydantic
5. JWT
6. Uvicorn

选择原因：

1. Python 更适合接入情绪识别、人脸检测和图片处理。
2. FastAPI 轻量、开发效率高，适合前后端分离接口开发。
3. 情绪识别库如 DeepFace、FER、OpenCV 等都能直接在 Python 中调用。
4. 相比 Spring Boot，FastAPI 更适合本项目的 AI 识别模块。

### Q8：数据库使用什么？

数据库暂定使用 MySQL。

ORM 推荐使用 SQLAlchemy。

选择原因：

1. MySQL 稳定常见，适合课程 / 毕设项目。
2. 数据结构清晰，便于保存用户、情绪记录、预警记录和视频分析记录。
3. 后续可以用 Navicat、DataGrip 等工具查看数据。

### Q9：情绪识别模块使用什么方案？

情绪识别模块推荐采用本地开源库，不优先使用付费 API。

候选方案：

1. DeepFace
2. FER
3. OpenCV + 预训练表情模型

第一版建议优先尝试 DeepFace 或 FER。

选择原因：

1. 不依赖昂贵付费接口。
2. 可以本地运行，适合课程项目演示。
3. 接入 FastAPI 后端比较自然。
4. 能够输出常见情绪类别，如 happy、sad、angry、fear、neutral 等。

如果本地模型环境安装不稳定，可以保留情绪识别服务抽象接口，后续切换其他模型或 API。

### Q10：项目环境配置怎么管理？

所有环境相关配置（数据库连接、JWT 密钥、模型路径等）统一放在后端 `.env` 文件中，不硬编码在代码里。

示例 `.env` 文件：
```
DATABASE_URL=mysql+pymysql://root:password@localhost:3306/mental_health
JWT_SECRET=your-secret-key
JWT_EXPIRE_HOURS=24
MODEL_NAME=deepface
UPLOAD_DIR=./uploads/temp
VIDEO_DIR=./vedio
CORS_ORIGINS=http://localhost:5173
```

后端使用 `python-dotenv` 或 Pydantic Settings 读取配置。这样换环境时只改 `.env` 文件，不需要改代码。

## 五、系统总体架构

### Q11：系统整体架构是什么？

系统采用前后端分离架构。前后端通过 HTTP 接口通信，后端需配置 CORS 允许前端域名跨域访问（开发环境前端为 `localhost:5173`，后端为 `localhost:8000`）。

前端负责：

1. 页面展示
2. 用户登录注册交互
3. 摄像头拍照 / 图片上传
4. 视频播放与抽帧
5. 情绪结果展示
6. 记录列表展示
7. 统计图表展示

后端负责：

1. 用户管理
2. 登录鉴权
3. 图片接收
4. 情绪识别（图片 + 视频帧）
5. 风险评分
6. 数据查询
7. 预警生成
8. 视频会话管理

数据库负责：

1. 保存用户数据
2. 保存情绪识别记录
3. 保存风险预警记录
4. 保存干预建议记录
5. 保存视频分析会话和帧级情绪记录

### Q11：一次情绪识别的流程是什么？

1. 学生登录系统。
2. 前端进入情绪识别页面。
3. 学生选择摄像头拍照或上传本地图片。
4. 前端将图片通过接口上传到后端。
5. 后端接收图片并进行基础校验。
6. 后端调用本地情绪识别模型。
7. 模型返回主要情绪、置信度和各情绪概率。
8. 后端根据情绪结果生成初步风险判断。
9. 后端保存识别记录到 MySQL。
10. 前端展示情绪识别结果和系统提示。

### Q12：是否保存用户上传的人脸原图？

第一版建议不长期保存人脸原图。

推荐策略：

1. 图片仅用于当次识别。
2. 后端识别完成后只保存结构化结果。
3. 数据库保存主要情绪、置信度、风险等级和识别时间。
4. 不保存或长期展示原始人脸图片。

这样可以降低隐私风险，也方便在计划书中说明系统的数据安全设计。

### Q15：情绪识别模型如何加载？

模型在 FastAPI 服务启动时**预加载一次**到内存，后续每次请求直接使用已加载的模型，避免每次识别都重新加载（DeepFace 每次加载约 3-10 秒）。

```python
# 服务启动事件
@app.on_event("startup")
async def load_model():
    from deepface import DeepFace
    app.state.model = DeepFace  # 模型常驻内存
```

纯 CPU 环境下单次识别耗时 2-5 秒，前端需显示 loading 状态或进度提示。

### Q16：上传的图片怎么处理？

1. 前端将图片通过 FormData 上传到后端
2. 后端保存到临时目录（如 `./uploads/temp/`），使用 UUID 重命名避免冲突
3. 调用情绪识别模型读取临时文件
4. 识别完成后**立即删除临时文件**
5. 只将结构化识别结果写入数据库

这样既满足识别需要，又不长期留存人脸图片。

### Q17：视频分析模块怎么设计？

管理员端提供视频分析页面，模拟摄像头采集过程，对预录视频进行逐帧情绪分析。

核心流程：

1. 管理员选择目标学生和预录视频文件（存放于 `./vedio/` 目录）
2. 设置抽帧数量（默认 20 帧，可调 5-60）
3. 点击"打开摄像头，开始采集数据"，系统模拟摄像头启动延迟（约 1.5s）
4. 黑框中播放预录视频（模拟"摄像头画面"）
5. 播放过程中按等间隔自动抽帧（间隔 = 视频时长 / 帧数）
6. 每帧通过 `<canvas>` 截取为 JPEG，立即发送后端进行情绪识别
7. 分析结果实时追加到下方表格和 ECharts 情绪变化曲线
8. 视频播放完毕后，后端汇总所有帧数据：主导情绪、负面占比、平均分数、情绪分布
9. 调用 `evaluate_video_risk()` 生成风险评估和干预建议
10. 中高风险自动创建预警记录

技术实现：
- 前端使用 `<video>` + 隐藏 `<canvas>` 进行帧截取
- 复用 `emotion_svc.analyze_face()` 进行单帧分析
- 复用 `risk.evaluate_video_risk()` 进行汇总评估
- 数据库新增 `video_analysis_sessions` 和 `video_frame_emotion_records` 两张表

### Q18：风险预警模块怎么设计？

风险预警采用规则评分方式。

评分来源包括：

1. 最近负面情绪出现次数
2. 是否连续多次出现负面情绪
3. 最近一次识别的主要情绪
4. 视频分析中的负面情绪占比和连续负面帧统计

负面情绪暂定包括：

1. sad
2. angry
3. fear
4. disgust

风险等级分为：

1. 低风险
2. 中风险
3. 高风险

### Q19：干预建议模块怎么设计？

第一版采用规则模板生成建议。

示例：

1. 低风险：建议保持规律作息，适当运动，继续观察。
2. 中风险：建议主动放松压力，与朋友或老师沟通。
3. 高风险：建议管理员重点关注，并建议学生及时寻求专业心理老师帮助。

系统生成的建议只作为辅助提醒，不替代专业心理咨询。

## 八、接口统一规范

### Q22：前后端接口统一用什么格式？

所有后端接口统一使用以下响应格式，避免前后端联调时格式不一致导致返工。

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

- `code`：业务状态码，200 表示成功
- `message`：提示信息，成功或错误描述
- `data`：实际返回数据，可以是对象、数组或 null

常见业务状态码：
| code | 含义 |
|------|------|
| 200 | 成功 |
| 400 | 请求参数错误 |
| 401 | 未登录或 Token 过期 |
| 403 | 权限不足 |
| 404 | 资源不存在 |
| 500 | 服务器内部错误 |

前端 Axios 统一拦截 `code` 做处理（如 401 跳转登录页），各页面只关注 `data`。

### Q26：系统需要测试哪些内容？

#### 登录注册测试

1. 学生可以正常注册。
2. 学生和管理员可以正常登录。
3. 密码错误时返回提示。
4. 未登录用户不能访问受保护接口。
5. 学生不能访问管理员接口。

#### 情绪识别测试

1. 上传正常人脸图片能够返回情绪结果。
2. 摄像头拍照后能够上传识别。
3. 上传非图片文件时返回错误。
4. 上传无人脸图片时返回友好提示。
5. 识别结果能够保存到数据库。

#### 视频分析测试

1. 能够列出 `./vedio/` 目录下的视频文件。
2. 能够通过流式接口播放视频。
3. 视频播放期间能够按间隔成功截取帧并发送分析。
4. 帧分析结果能够实时出现在前端表格和图表中。
5. 会话完成后能够生成正确的汇总评估。
6. 历史会话列表能够正确展示和查看详情。

#### 预警测试

1. 负面情绪次数较多时触发预警。
2. 视频分析负面占比较高时触发预警。
3. 管理员能够查看预警列表。
4. 管理员能够更新预警处理状态。

#### 可视化测试

1. 学生能够查看个人情绪趋势。
2. 管理员能够查看情绪分布。
3. 管理员能够查看风险等级统计。

## 十三、项目风险与应对

### Q27：项目可能遇到哪些问题？

#### 1. 情绪识别库安装失败

应对方式：

1. 优先使用 DeepFace。
2. 如果环境依赖较复杂，可切换 FER。
3. 如果仍不稳定，可临时使用模拟识别结果完成系统闭环。

#### 2. 识别准确率不高

应对方式：

1. 在计划书中明确系统为辅助监测。
2. 不将识别结果作为医学诊断。
3. 结合长期趋势和多次分析降低单次识别误差影响。

#### 3. 隐私风险

应对方式：

1. 不长期保存人脸原图。
2. 密码加密存储。
3. 使用角色权限控制。
4. 管理员只能查看必要信息。

#### 4. 开发时间不足

应对方式：

1. 先完成 P0。
2. P1 完成趋势、记录和视频分析。
3. P2 中的数据大屏和高级建议可作为扩展。
